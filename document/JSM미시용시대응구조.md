# 🚧 JSM 미사용 시 고객 문의 대응 구조 (기본 Jira 템플릿 기반)

---

### 🧩 1. 개요

발주사 요청에 따라 **Jira Service Management(JSM)** 을 사용하지 않고

**기본 Jira Software 템플릿**만으로 고객 문의 대응 인터페이스를 구성할 수 있는지 검토한 결과입니다.

---

### 💡 2. 결론 요약

> ✅ 가능 — 기본 Jira API를 이용해 최소한의 문의 등록/상태 확인 인터페이스는 구현 가능
>
> ❌ **불편함 존재** — 고객 포털, 자동 알림, SLA 관리 등은 직접 구현 필요

---

### ⚙️ 3. 기본 Jira로 구현 가능한 항목

| 기능                | 구현 가능 여부 | 구현 방식                                                           |
| ------------------- | -------------- | ------------------------------------------------------------------- |
| 고객 문의 등록      | ✅             | AtSignal 웹사이트 → Jira REST API로 이슈 생성 (`/rest/api/3/issue`) |
| 이슈 상태 관리      | ✅             | Jira 워크플로우 사용 (예: 접수 → 진행중 → 완료)                     |
| 담당자 지정         | ✅             | 이슈 생성 시 `assignee` 필드 설정                                   |
| 상태 변경 알림      | ✅             | Jira Webhook → AtSignal 서버로 전달                                 |
| 문의 목록/상태 조회 | ✅             | Jira REST API를 통해 AtSignal에서 표시                              |

---

### ⚠️ 4. 기본 Jira의 한계 및 보완 필요 항목

| 항목                           | 기본 Jira 상태 | 보완 방안                                            |
| ------------------------------ | -------------- | ---------------------------------------------------- |
| 고객 포털 (문의 접수 UI)       | ❌ 없음        | AtSignal 웹사이트에서 직접 구현                      |
| 고객 이메일 알림               | ❌ 없음        | SMTP 또는 SendGrid API 연동 필요                     |
| 고객 권한 분리                 | ⚠️ 제한적      | 고객은 Jira 직접 접근 불가 → AtSignal 내부 UI로 대체 |
| SLA, 리포트                    | ❌ 없음        | 내부적으로 별도 통계 로직 구축 필요                  |
| 자동화 (예: 상태 변경 시 알림) | ⚙️ 제한적      | Webhook + 내부 서버 로직으로 구현 가능               |

---

### 🧱 5. 권장 구성 예시

```
[ 고객 ]
   ↓ (문의 등록)
[ AtSignal 웹사이트 ]
   ↓ REST API
[ Jira (기본 템플릿 프로젝트) ]
   ↓ Webhook
[ AtSignal 서버 ]
   ↓
[ 고객 알림 (웹 UI 또는 이메일) ]

```

---

### 🧭 6. 결론

- JSM 없이도 **최소한의 고객 문의 흐름**은 구현 가능
- 다만 **고객용 포털/알림 시스템은 전부 AtSignal 측에서 직접 구현해야 함**
- 장기적으로 고객응대 효율성과 유지보수를 고려하면 **JSM 도입이 권장됨**
