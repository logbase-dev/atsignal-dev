# 📘 AtSignal 다국어(i18n) 전략 및 구현 방식 정리

_(한글/영문 서비스 논의 내용 기반)_

---

# 📌 1. 다국어 구조에 대한 기본 방향성

AtSignal 웹사이트는 페이지 수가 많고(약 70개), 정적 페이지 + 동적 페이지가 혼합된 형태이기 때문에

**정적 번역(JSON) + 동적 번역(Firestore)** 조합이 가장 현실적인 구조로 판단됨.

---

# 🟦 2. 제안 방식 A: “한글 완성 → 영어 자동 생성” 단계적 확장 전략 (권장 ⭐)

### ✅ 개요

8주 개발 기간 동안 **한글 버전을 100% 완성**한 뒤,

출시 직전 또는 출시 직후 **영문을 자동 번역 기반으로 반자동 생성**하는 구조.

---

### 🧩 진행 단계

### ① **한글 페이지 100% 완성**

- 페이지 구조, UI, 라우팅, SEO, OG, Sitemap 모두 한글 기준으로 구축
- 모든 텍스트는 `locales/ko.json` 파일로 분리 (영문 대비 구조 확보)

### ② **영문 자동 생성**

- 빌드 파이프라인에서 자동 번역 API 사용
  - DeepL API
  - Google Translate API
  - 자동 번역 API 비용 문제
    - Google Translate API
    - 약 **$20 / 1,000,000 characters**
    - 보통 한글 전체 페이지가 5~50k 자라면
      → 1회 자동 번역 비용은 **$1 미만**
- `ko.json → en.json` 자동 생성 후
  일부 핵심 페이지만 수동 교정

### ③ **정식 영문 적용 (v1.1)**

- 런칭 후 1~2주 내 모든 영문 페이지 검수 완료

---

### ⚙ 기술 구조

- `/ko` → 수동 콘텐츠
- `/en` → 자동 생성된 콘텐츠 + “자동 번역 버전” 안내 문구 표시
- SEO 손실 없음 (OG 및 sitemap 미리 구성)

---

### 🎯 장점

- 번역 때문에 개발 일정 지연 없음
- 영어 페이지도 구조는 같기 때문에 유지보수 편함
- 런칭 후에도 빠르게 영문 확장 가능

---

# 🟩 3. 제안 방식 B: “하이브리드 i18n + 단일 빌드” 전략

### ✅ 개요

초기에는 **한국어만 완성**, 영문 텍스트는 `en.json`에 비워둠.

영문 URL로 접근하면 fallback으로 **한국어가 노출**되도록 구성.

---

### 🧩 진행 단계

### ① 모든 텍스트를 `t('key')`로 추출

- `locales/ko.json`만 실제 value 채움
- `locales/en.json`은 key만 존재, value는 비워두기

### ② `/en` 접근 시 fallback to Korean

- en.json에 value가 없으면
  → `ko.json`의 내용을 대신 표시
- “영문 페이지 준비 중” 안내 필요
- en.json 파일 자체가 없어도 되고, en.json에 header.contactCta가 없어도 ko.json에 있는 "문의하기" 값이 사용됨.

### ③ 이후에 번역된 en.json만 업데이트

- 기능 개발은 지연 없음
- 영문 번역은 개발 완료 후 천천히 진행 가능

---

### 🎯 장점

- 초기 개발 부담 최소화
- 언어 구조만 미리 잡아서 확장 용이
- URL 구조를 미리 확보하므로 SEO에도 유리

---

# 🟧 4. 정적/동적 콘텐츠 번역 방식 비교

## 🟦 정적 페이지(문서, 소개 페이지 등)

- `locales/ko.json`, `locales/en.json`에 번역 저장
- **빌드에 포함하는 것이 성능·안정성 최상**
- **재배포 필요**

---

## 🟩 동적 페이지(메뉴 자동 생성, CMS 기반 페이지)

- Firestore에 다음처럼 저장:

```
pages/
   solution-intro/
			{
			  "title": {
			    "ko": "행동 분석 솔루션",
			    "en": "Behavior Analytics Solution"
			  },
			  "body": {
			    "ko": "...",
			    "en": "..."
			  },
			  "autoTranslated": true  // 자동 번역된 상태인지 기록
			}
```

1. 저장 시 자동 번역 후 ko/en 모두 저장
2. 운영자가 ko/en 각각 입력

- 정적 번역과 관계없이 Firestore만 업데이트하면 반영됨

---

# 🟥 5. Storage에 JSON 파일 저장하는 방식은 비추천

### 🔻 이유

- 페이지 로딩 시 Storage에서 JSON fetch → **속도 매우 느림**
- 첫 페이지 로딩 시 지연(FCP 악화)
- 네트워크 불안하면 번역 안 뜨는 문제 발생
- 버전 관리 어려움
- SEO 문제 발생 가능
- 페이지 수가 많을수록 비효율적

### ✔ 결론

**정적 번역(JSON) → 빌드 포함
동적 번역 → Firestore 사용**

이 구조가 성능/운영 모두 가장 안정적.

---

# 🟦 6. 최종 권장 아키텍처 요약

| 항목             | 권장 방식                  | 이유                    |
| ---------------- | -------------------------- | ----------------------- |
| 정적 페이지 번역 | `locales/*.json` 빌드 포함 | 성능 최고, 구조 명확    |
| 동적 페이지 번역 | Firestore 저장             | 운영자 실시간 편집 가능 |
| 영어 자동 생성   | 빌드 파이프라인 자동 번역  | 일정 단축 + 품질 유지   |
| 영어 완성        | 런칭 이후 순차적 검수      | 리스크 최소화           |
| 영문 페이지 모드 | `/en` URL 유지             | SEO 확보                |
